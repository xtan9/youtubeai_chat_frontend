# Traditional Chinese (Taiwan) ‚Äî output language support

**Date:** 2026-05-01
**Status:** Approved (verbal, brainstorm session 2026-05-01)

## Summary

Add `zh-TW` ("Chinese (Traditional, Taiwan)") as a user-selectable
**output language** for video summaries, alongside the existing 17
codes. Browser-language detection routes Traditional-script readers
(`zh-TW`, `zh-HK`, `zh-MO`, `zh-Hant*`) to the new `zh-TW` summary;
all other `zh-*` tags continue to default to existing `zh`
(Simplified). The summarization prompt, picker UI, cache layer, and
chat path all pick up the new code with no further changes ‚Äî the
addition slots into the existing per-language pipeline that already
ships `es`, `pt`, `it`, `fr`, `de`, `id`, `zh`, `ja`, `ko`, `ar`,
`hi`, `bn`, `ru`, `vi`, `tr`, `th`.

This change is **output-only**. It does not introduce a UI-i18n
framework, locale routing on the marketing pages, or a Traditional
translation of any chrome (buttons, nav, errors). Those are
explicitly out of scope per the brainstorm.

## Background

The existing summary pipeline supports an `output_language` query
parameter (`/api/summarize/stream?output_language=es`, etc.) that
threads through to:

- `LanguagePicker` UI (`app/summary/components/language-picker.tsx`)
  ‚Äî auto-renders any code present in `SUPPORTED_OUTPUT_LANGUAGES`.
- Summarization prompt (`lib/prompts/summarization.ts`) ‚Äî injects
  `Respond in ${getLanguage(code).english}.` so a new entry needs no
  prompt-code change, only a constants entry.
- Per-language summary cache (`lib/services/summarize-cache.ts`) ‚Äî
  rows are keyed by `(video_id, output_language)`. A new code creates
  a new partition; existing `zh` rows are not invalidated and
  Simplified users keep getting cache hits.
- Chat (`app/api/chat/stream/route.ts`, `lib/prompts/chat.ts`) ‚Äî
  already says "Match the language of my question." so no change
  needed.
- Hero demo registry (`app/components/hero-demo-data/index.ts`) ‚Äî
  per-(video, language) static loaders, generated by
  `scripts/build-hero-demo-data.ts`.

Browser-language detection
(`lib/utils/browser-locale.ts::pickDefaultLanguage`) currently
matches on the **primary subtag only** (`zh-TW` ‚Üí `zh`), so adding
`zh-TW` to the supported set requires the matcher to learn about
script and region subtags or it will silently route Taiwan readers
to Simplified.

## Browser matcher: alias-table design

```ts
// lib/utils/browser-locale.ts (after the change)

import type { SupportedLanguageCode } from "@/lib/constants/languages";

// BCP-47 tag ‚Üí supported code, applied before the primary-subtag fallback.
// Keep the keys lowercase to match the lowercased navigator tag.
//
// Why an alias table instead of inline conditionals: the next regional
// locale we add (e.g. pt-BR vs pt) becomes a one-line addition rather
// than another nested if-statement.
const TAG_ALIASES: Readonly<Record<string, SupportedLanguageCode>> = {
  "zh-tw": "zh-TW",
  "zh-hk": "zh-TW",
  "zh-mo": "zh-TW",
  "zh-hant": "zh-TW",
};

export function pickDefaultLanguage(
  navigatorLanguages: readonly string[],
  supported: readonly SupportedLanguageCode[]
): SupportedLanguageCode {
  const supportedSet = new Set<string>(supported);
  for (const raw of navigatorLanguages) {
    if (!raw) continue;
    const tag = raw.toLowerCase();

    // 1. Exact match (e.g. "zh-tw" ‚Üí "zh-TW" when supported set has "zh-TW").
    //    Walk the supported set case-insensitively.
    const exact = supported.find((c) => c.toLowerCase() === tag);
    if (exact) return exact;

    // 2. Alias match by full tag (e.g. "zh-hk" ‚Üí "zh-TW").
    if (TAG_ALIASES[tag]) return TAG_ALIASES[tag];

    // 3. Alias match by script-stripped tag (e.g. "zh-hant-tw" ‚Üí "zh-TW"
    //    via "zh-hant"). Try progressively shorter prefixes so
    //    "zh-hant-cn" hits the "zh-hant" alias rather than collapsing to
    //    "zh".
    const parts = tag.split("-");
    for (let len = parts.length - 1; len >= 2; len--) {
      const prefix = parts.slice(0, len).join("-");
      if (TAG_ALIASES[prefix]) return TAG_ALIASES[prefix];
    }

    // 4. Primary-subtag fallback (existing behavior). Catches "zh-cn" ‚Üí "zh",
    //    "en-US" ‚Üí "en", "es-419" ‚Üí "es", etc.
    const primary = parts[0];
    if (supportedSet.has(primary)) {
      return primary as SupportedLanguageCode;
    }
  }

  // 5. Final fallback. English is safe: it's in the supported set and every
  //    user can read an English label in the dropdown.
  return "en";
}
```

## Hero demo data ‚Äî bootstrap and follow-up

The hero demo loader matrix is
`Record<HeroDemoVideoId, Record<SupportedLanguageCode, ‚Ä¶>>`, so adding
`zh-TW` to the constants makes `tsc` require a `zh-TW.ts` file for
each of the 6 video ids. The compile-time exhaustiveness guard is
the existing pattern; we will not relax it.

The seed‚Üídump‚Üíbuild pipeline that produces real per-language demo
content needs **prod-side** support for the new code (the
`/api/summarize/stream` zod enum rejects unknown codes), and
`youtubeai_chat_frontend/.env.local` doesn't carry the LLM gateway
credentials, so we can't generate real `zh-TW` summaries from this
session.

**Bootstrap path (this PR):** ship 6 placeholder `zh-TW.ts` files
that copy the existing `zh.ts` (Simplified) content into the new
slot, with `language: "zh-TW"`. The loader matrix compiles, the
homepage hero demo widget renders something for `zh-TW` visitors,
and the file shape exactly matches what the build script will
overwrite later. Each placeholder file carries a single
`// PLACEHOLDER:` comment naming the follow-up plan.

**User-visible impact of placeholder:** a zh-TW-locale visitor on
the homepage may see Simplified characters in the small demo widget
until the follow-up data refresh lands. The real picker ‚Üí real
summary flow works correctly from day one (server-side cache misses
generate genuine zh-TW content via Claude on first request).

**Follow-up (data-only PR, post-deploy):**

1. After this PR ships and prod accepts `zh-TW`, run
   `set -a; source ~/.config/claude-test-creds/youtubeai.env; set +a`
   then `pnpm tsx scripts/seed-hero-demo-translations.ts`.
   The 102 existing combos return `cached`; the 6 new `zh-TW`
   combos generate fresh.
2. Dump the prod cache into `/tmp/yt-demo-data/all.json` (per the
   SQL dump pipeline referenced in `scripts/build-hero-demo-data.ts`).
3. `pnpm tsx scripts/build-hero-demo-data.ts` overwrites the 6
   placeholder files with real `zh-TW` summaries.
4. Open a small data-only PR with the regenerated files.

The follow-up is on the order of 6 LLM calls and a few minutes of
local script runtime.

## Cache compatibility

The `summaries` table's `output_language` column is unconstrained
text (the application enforces the enum at the API edge). New
`zh-TW` rows do not collide with existing `zh` rows. Existing
Simplified users see no behavior change. Until a video is requested
in `zh-TW`, no `zh-TW` row exists ‚Äî first request pays the
generation cost, subsequent requests hit the cache.

No database migration is required. No backfill is required.

## Out of scope

- UI chrome translation (buttons, nav, error messages, toasts).
- Marketing-page localization (landing, pricing, FAQ, blog).
- `hreflang` tags or `/zh-TW/...` URL routing.
- Legal page (privacy, terms) translation.
- Hong Kong / Macau-specific Traditional output (HK Cantonese-style
  vocabulary differs from Taiwan-style; we route both to one Taiwan
  variant by design).
- Server-side `Accept-Language` header reading (the picker remains
  client-side, hydrated from `navigator.languages`).
- Changing `PromptLocale` from binary `en|zh` (the
  routing-classifier locale; orthogonal to output language).
