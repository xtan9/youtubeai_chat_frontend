# Navigation

The navigation cluster groups the seven components used to move users
through the app: top-level site nav, application menu bars, in-page
view switching, hierarchical position, paginated lists, command
palettes, and per-trigger action menus. Each component ships a Vitest
behavior test, a jest-axe a11y test (zero violations across every
variant + edge state), and a per-component MDX doc with prop tables,
variants, and composition examples.

## Members

### Top-level navigation

- [NavigationMenu](../navigation-menu.mdx) — site-header strip with
  optional drop-down panels per item. Backed by
  `@radix-ui/react-navigation-menu`. Reach for it on a marketing
  header where each item may expand into a richer panel.
- [Menubar](../menubar.mdx) — application-style menu bar (File / Edit
  / View). Backed by `@radix-ui/react-menubar`. Reach for it when
  several distinct menus need to live side by side with roving focus
  between them.
- [Breadcrumb](../breadcrumb.mdx) — hierarchical trail of links to
  ancestors. Pure semantic HTML.

### In-page navigation

- [Tabs](../tabs.mdx) — switch between mutually exclusive panels
  sharing a surface. Backed by `@radix-ui/react-tabs`.
- [Pagination](../pagination.mdx) — page-by-page navigation through a
  list. Pure semantic HTML; visual delegates to
  [Button](../button.mdx).

### Action menus

- [DropdownMenu](../dropdown-menu.mdx) — single trigger button
  exposing a list of actions. Backed by
  `@radix-ui/react-dropdown-menu`.
- [Command](../command.mdx) — keyboard-first command palette /
  fuzzy-search list. Backed by [`cmdk`](https://github.com/pacocoursey/cmdk).

## When to reach for which

The choice depends on **where** the navigation lives, **what** it
moves between, and **how** the user gets to it:

| Scenario                                                                   | Reach for                              |
| -------------------------------------------------------------------------- | -------------------------------------- |
| Site header on a marketing page; products / resources need rich panels     | [NavigationMenu](../navigation-menu.mdx) |
| Application top bar with multiple menus (File / Edit / View)               | [Menubar](../menubar.mdx)              |
| Hierarchical trail (Home › Library › Playlist)                             | [Breadcrumb](../breadcrumb.mdx)        |
| Switch between sub-views of one page (Account / Password / Notifications)  | [Tabs](../tabs.mdx)                    |
| Paginated list (page 1 / 2 / … / 24)                                       | [Pagination](../pagination.mdx)        |
| One trigger button exposing several actions (overflow `…`, avatar)         | [DropdownMenu](../dropdown-menu.mdx)   |
| Keyboard-first "type then pick" UX (⌘K palette, quick switcher)            | [Command](../command.mdx)              |

## DropdownMenu vs Menubar — close call

Both wrap the same Radix menu primitive family. The shape of the
trigger surface decides:

- **DropdownMenu**: one trigger button. Click opens a single menu.
- **Menubar**: a strip of triggers with roving focus between them.
  Open one, arrow left/right hops to its neighbour without an extra
  click. Use this when the user expects desktop-app menu ergonomics.

## NavigationMenu vs Menubar — also close

Both are persistent strips, but:

- **NavigationMenu** is meant for the marketing/site `<nav>` landmark.
  Items are usually links (or links nested in a panel) — semantically
  navigation, not action.
- **Menubar** is for application chrome where each menu lists
  *actions* (commands), not navigation targets.

When in doubt: are the leaf items routes you can bookmark?
NavigationMenu. Or commands the user is invoking? Menubar.

## Accessibility patterns

- **NavigationMenu**, **Menubar**, **DropdownMenu**, **Command** all
  use Radix focus guards around the open overlay; the cluster a11y
  tests scan with `axeOverlay` (or `axePortalOverlay` for
  DropdownMenu/Menubar, which also portal outside any landmark).
- **Command** uses `cmdk`, which renders its separator with
  `role="separator"` directly inside the listbox. axe's
  `aria-required-children` rule (correctly) flags this — the cluster
  uses an `axeCommand` runner that suppresses that one rule with the
  rationale documented in `tests-utils/axe.ts`.
- **Breadcrumb** marks the trail terminus with `aria-current="page"`;
  intermediate links are normal `<a>`s.
- **Pagination** marks the active page with `aria-current="page"`;
  Previous / Next get explicit `aria-label`s for icon-only mobile
  layouts.
- **Tabs** auto-wires `aria-controls` / `aria-labelledby` between
  triggers and panels. Pass `aria-label` on `TabsList` so AT users
  hear a name for the strip.
- **Disabled triggers / items / tabs** are skipped during arrow-key
  navigation; their `data-disabled=""` attribute is read by axe as
  `aria-disabled`.

## Tested via

- `components/ui/__tests__/<name>.test.tsx` — behavior (rendering,
  controlled / uncontrolled, keyboard + pointer interactions,
  data-slot wiring, edge states).
- `components/ui/__tests__/<name>.a11y.test.tsx` — jest-axe (zero
  violations across variants + edge states).
- Shared helpers (`tests-utils/`):
  - `renderWithProviders.tsx` — wraps with ThemeProvider + automatic
    cleanup.
  - `axe.ts` exports the configured runners:
    - `axe` — default (WCAG 2.1 A + AA, all rules).
    - `axeOverlay` — disables `aria-hidden-focus` for Radix focus
      guards (Dialog, NavigationMenu open viewport, etc.).
    - `axePortal` — disables `region` for portaled non-landmark
      overlays (Tooltip, HoverCard).
    - `axePortalOverlay` — disables both, used for Radix menu
      family that portals outside any landmark (DropdownMenu,
      Menubar).
    - `axeCommand` / `axeCommandDialog` — disables
      `aria-required-children` for the cmdk separator; the latter
      also disables `aria-hidden-focus` for the wrapping Dialog.

## Happy-dom interaction notes

The Radix menu primitive family (DropdownMenu, Menubar) opens on a
complete `pointerdown → pointerup → click` sequence. happy-dom
doesn't synthesize pointer events from `fireEvent.click()`, so the
cluster tests use a small `activate(el)` helper that fires the
sequence directly. Tabs activate on bare `mousedown` — the tabs
test fires it directly.
